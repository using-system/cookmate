# Android APK on GitHub Release (Phase 1) — Design

**Date:** 2026-04-18
**Status:** Approved

## Context

The current release workflow (`.github/workflows/release.yaml`) computes the
next semantic version from Conventional Commits and creates a GitHub Release
with auto-generated notes. No build artifact is attached, so installing the app
on a real device requires a local Flutter toolchain.

This spec is **Phase 1** of a two-phase distribution plan. Phase 1 ships a
signed Android APK as an asset of every GitHub Release, so the maintainer can
install or update the app on any Android device by visiting the latest release
URL from the device browser.

Phase 2 (separate spec, later) will add Firebase App Distribution as a second
delivery channel. Phase 2 will reuse the APK produced in Phase 1.

## Goals

- Produce a signed `release` APK on every push to `main` that bumps the
  semantic version.
- Attach the APK as an asset of the GitHub Release created by the existing
  workflow.
- Make the APK installable across Android devices and updatable in-place
  (stable signature) without uninstalling the previous version.
- Keep local development working without any signing setup (debug fallback).
- Keep the signing key material out of the repository.

## Non-Goals

- No Play Store publication (no `.aab`).
- No Firebase App Distribution (Phase 2).
- No iOS build, no TestFlight, no IPA distribution.
- No automatic version bumping inside `pubspec.yaml`.
- No multi-flavor build (a single `release` APK is enough).
- No split-per-ABI APKs (a single universal APK is shipped).
- No Flutter test or analyze steps inside the release workflow (CI workflow
  already covers that on every PR and on `main`).

## Tech Stack

- **GitHub Actions** — same runner type (`ubuntu-latest`) and Flutter version
  (`3.41.7`) as the existing CI workflow.
- **`subosito/flutter-action`** — already used by CI; reused here for the
  Android build job.
- **`actions/upload-artifact` / `actions/download-artifact`** — to pass the
  APK from `build-android` to `release`.
- **`softprops/action-gh-release`** — already used by `release`; gains the
  `files:` parameter to attach the APK.
- **Gradle signing config** in `android/app/build.gradle.kts` — reads an
  optional `key.properties` file generated by the workflow at build time.

## Architecture

### Job graph

```
version ──► build-android ──► release
   │              │              │
   │              │              └─ create GitHub Release vX.Y.Z
   │              │                 + attach cookmate-vX.Y.Z.apk
   │              │
   │              └─ build signed APK
   │                 upload artifact (cookmate-apk)
   │
   └─ compute new_tag from Conventional Commits
```

All three jobs share the same gate: they only run their effective work when
`needs.version.outputs.new_tag != ''`. A push without a `feat:` / `fix:`
commit produces no tag, no APK, no release.

### `build-android` job

1. Checkout (no `fetch-depth: 0` needed — only the working tree is built).
2. Setup Flutter (same action and version as CI).
3. Reconstitute signing material from secrets:
   - Decode `ANDROID_KEYSTORE_BASE64` to `android/upload-keystore.jks`.
   - Write `android/key.properties` with `storeFile=upload-keystore.jks`,
     `storePassword`, `keyAlias`, `keyPassword` from the corresponding
     secrets.
4. `flutter pub get`.
5. `flutter build apk --release --build-name=${TAG#v} --build-number=${GITHUB_RUN_NUMBER}`.
6. Rename `build/app/outputs/flutter-apk/app-release.apk` to
   `cookmate-${TAG}.apk`.
7. Upload as workflow artifact (`cookmate-apk`, retention 7 days).

### `release` job (modified)

1. Add `build-android` to `needs`.
2. Download the `cookmate-apk` artifact.
3. Pass the APK path through `softprops/action-gh-release`'s `files:` input.

### Gradle signing wiring

`android/app/build.gradle.kts` is updated to:

- Load `android/key.properties` if it exists.
- Define a `release` signing config that reads `storeFile`, `storePassword`,
  `keyAlias`, `keyPassword` from that file.
- Set `buildTypes.release.signingConfig` to the `release` config when
  `key.properties` is present, otherwise fall back to `signingConfigs.debug`
  (so `flutter run --release` keeps working locally without any setup).

`android/key.properties` and `android/upload-keystore.jks` are already covered
by the existing `.gitignore` entries (`**/android/key.properties` and
`*.jks`). Both are created by the CI workflow at build time and never
committed.

### Versioning

`pubspec.yaml` stays at its current value. The APK version is overridden at
build time:

- `--build-name=${TAG#v}` → human-readable version shown in
  Settings → Apps → Cookmate (e.g. `1.1.0`).
- `--build-number=${GITHUB_RUN_NUMBER}` → integer monotonic across workflow
  runs, required by Android to accept an in-place update.

### Discovery from a mobile device

The maintainer opens
`https://github.com/using-system/cookmate/releases/latest` from Chrome on the
target phone, taps `cookmate-vX.Y.Z.apk`, and Android offers to install or
update.

## Secrets

The four required secrets are already provisioned at the repository level
(`using-system/cookmate`):

- `ANDROID_KEYSTORE_BASE64` — base64 of the PKCS12 upload keystore.
- `ANDROID_KEYSTORE_PASSWORD` — keystore password.
- `ANDROID_KEY_ALIAS` — key alias inside the keystore (`cookmate-upload`).
- `ANDROID_KEY_PASSWORD` — equal to `ANDROID_KEYSTORE_PASSWORD` because the
  keystore is in PKCS12 format (which does not support distinct store and
  key passwords).

The keystore itself lives outside the repository on the maintainer's machine
at `~/.cookmate/upload-keystore.jks`. It is not regenerated by CI.

## Files Touched

| File | Change |
|---|---|
| `.github/workflows/release.yaml` | Add `build-android` job; extend `release` job with `needs: build-android`, artifact download, and `files:` on `softprops/action-gh-release`. |
| `android/app/build.gradle.kts` | Conditional `release` signing config reading `key.properties`; debug fallback when absent. |
| `pubspec.yaml` | Unchanged. |
| `.gitignore` | Unchanged (already covers `*.jks` and `**/android/key.properties`). |

## Risks and Mitigations

- **Signature mismatch with previously installed debug build.** Any device
  that already has the current debug-signed APK installed must uninstall it
  before installing the first signed release APK. One-time cost per device.
- **Public repository.** The APK is downloadable by anyone visiting the
  Releases page. Acceptable for an open-source project; flagged here to
  avoid future surprise.
- **Workflow runtime.** The Android build adds roughly 3–5 minutes to the
  release workflow. Acceptable since it only runs on releasable commits.
- **Loss of the local `.jks`.** The base64 copy lives in GitHub secrets, so
  CI keeps signing successfully. Recovering the raw `.jks` for local use
  would require a one-off workflow that re-emits the secret. Documented but
  not automated in Phase 1.

## Validation

This is an infrastructure change with no Flutter test impact. Validation is
manual, performed once after merge:

1. Land a `feat:` or `fix:` commit on `main`.
2. Confirm the `version` job produces a new tag `vX.Y.Z`.
3. Confirm the `build-android` job uploads `cookmate-vX.Y.Z.apk`.
4. Open `https://github.com/using-system/cookmate/releases/latest`,
   verify the APK is attached, download it on an Android device, install
   it, and verify the version visible in Android Settings matches `X.Y.Z`.
5. Push a second qualifying commit; verify the next APK installs in-place
   over the previous one without requiring uninstall.
